1. PRODUCT VISION & MISSION
================================================================================

PRODUCT NAME:    StockPilot
TAGLINE:         AI-Powered Autonomous Inventory Management Platform

--------------------------------------------------------------------------------
1.1 VISION STATEMENT
--------------------------------------------------------------------------------

StockPilot aims to become the operating system for modern warehouse operations.
It replaces fragmented, manual systems such as spreadsheets and paper logs with
a unified, intelligent platform that gives warehouse teams real-time visibility,
predictive intelligence, and the ability to control inventory using natural
language.

The platform eliminates guesswork in stock management by combining real-time
tracking with machine learning forecasting, automation rules, and a conversational
AI assistant — enabling businesses to reduce stockouts, minimize overstock, and
optimize warehouse space utilization.

--------------------------------------------------------------------------------
1.2 MISSION
--------------------------------------------------------------------------------

To make inventory management as simple as having a conversation, as intelligent
as a data scientist, and as reliable as an enterprise ERP — for businesses of
all sizes.

--------------------------------------------------------------------------------
1.3 CORE VALUE PROPOSITIONS
--------------------------------------------------------------------------------

a) Real-Time Visibility       — Know exactly what stock exists, where it sits,
                                and how it is moving at all times.

b) AI-Driven Decisions        — Predictive demand forecasting and reorder
                                suggestions powered by machine learning.

c) Natural Language Control   — Execute complex inventory operations by typing
                                plain English commands.

d) Automation                 — Rules-based background workers handle reorder
                                triggers, slow-moving item detection, and alerts
                                without human intervention.

e) Visual Warehouse Layout    — Interactive grid map of every rack, shelf, and
                                zone with color-coded utilization indicators.

f) Audit Trail                — Every stock movement, command, and automation
                                action is logged with timestamps and user context.

3. TECH STACK & TECHNOLOGY INTERACTIONS
================================================================================

--------------------------------------------------------------------------------
3.1 TECHNOLOGY COMPONENTS
--------------------------------------------------------------------------------

FRONTEND:
  Framework:       Next.js 14 (App Router)
  UI Library:      React 18
  Styling:         TailwindCSS 3
  Component Kit:   Shadcn UI (built on Radix UI primitives)
  State Mgmt:      Zustand or React Context
  Data Fetching:   SWR / React Query
  Charts:          Recharts or Nivo
  Icons:           Lucide React

BACKEND:
  Runtime:         Next.js API Routes (serverless functions on Vercel)
  Language:        TypeScript
  ORM/DB Client:   Supabase JS Client
  Auth Middleware: Supabase Auth helpers for Next.js
  Background Jobs: BullMQ (Redis-backed) on Render/Railway

DATABASE & AUTH:
  Provider:        Supabase
  Engine:          PostgreSQL 15
  Auth:            Supabase Auth (JWT-based, email/password)
  Storage:         Supabase Storage (for any file attachments)
  Realtime:        Supabase Realtime subscriptions for live dashboard updates

AI ASSISTANT:
  Provider:        Grok API (xAI)
  Model:           grok-beta or grok-2
  Architecture:    RAG (Retrieval-Augmented Generation)
  Vector Store:    FAISS (Python service)
  Embeddings:      Sentence Transformers (all-MiniLM-L6-v2)

ML PREDICTION SERVICE:
  Language:        Python 3.11
  Framework:       FastAPI
  Libraries:       scikit-learn, Prophet (Facebook), pandas, numpy
  Deployment:      Render (background service)

VECTOR DATABASE:
  Engine:          FAISS (Facebook AI Similarity Search)
  Purpose:         Store embeddings of inventory records, logs, and product data
                   for semantic search by the AI assistant

DEPLOYMENT:
  Frontend:        Vercel (automatic CI/CD from GitHub)
  Backend APIs:    Vercel serverless (co-located with Next.js)
  ML Service:      Render (Python FastAPI container)
  Job Workers:     Railway or Render (Node.js BullMQ workers)
  Database:        Supabase (managed PostgreSQL)
  Cache/Queue:     Redis via Railway

--------------------------------------------------------------------------------
3.2 HOW TECHNOLOGIES INTERACT
--------------------------------------------------------------------------------

REQUEST FLOW — USER QUERIES INVENTORY:

  [Browser/Next.js App]
       |
       | HTTP Request
       v
  [Next.js API Route] (/api/inventory)
       |
       | Supabase JS Client
       v
  [Supabase PostgreSQL]
       |
       | Returns rows
       v
  [Next.js API Route] — formats JSON response
       |
       v
  [React Component] — renders table/chart

REQUEST FLOW — AI ASSISTANT QUERY:

  [User types question in AI Chat]
       |
       v
  [Next.js API Route] (/api/ai/query)
       |
       |--- Supabase: fetch relevant inventory context (recent moves, stock levels)
       |--- FAISS Service: retrieve semantically similar historical records (RAG)
       |
       v
  [Grok API] — receives context + question → generates response
       |
       v
  [AI Assistant UI] — streams response to user

REQUEST FLOW — AUTOMATION RULES:

  [BullMQ Cron Worker] — runs every 15 minutes
       |
       | Reads automation_rules table
       v
  [Rule Evaluator] — checks conditions against live Supabase data
       |
       | If condition met:
       v
  [Action Executor] — creates alert / sends notification / marks item
       |
       v
  [ai_action_logs table] — records execution for audit

REQUEST FLOW — DEMAND PREDICTION:

  [Operations Manager requests forecast]
       |
       v
  [Next.js API] → calls Python FastAPI service on Render
       |
       v
  [Prophet/scikit-learn model] — loads historical delivery data from Supabase
       |
       v
  [Returns forecast JSON] → displayed as chart in dashboard

4. SYSTEM ARCHITECTURE
================================================================================

--------------------------------------------------------------------------------
4.1 LAYER OVERVIEW
--------------------------------------------------------------------------------

LAYER 1: FRONTEND (Presentation Layer)
  - Next.js App Router with server and client components
  - Shadcn UI components for consistent design system
  - TailwindCSS for utility-first styling
  - Supabase Realtime for live updates on dashboard widgets
  - Client-side state management for optimistic UI updates

LAYER 2: API LAYER (Next.js API Routes)
  - RESTful API endpoints under /api/*
  - Middleware for JWT validation via Supabase Auth
  - Role-based access control (RBAC) enforcement
  - Request validation using Zod schemas
  - Rate limiting per user/IP

LAYER 3: DATABASE LAYER (Supabase / PostgreSQL)
  - PostgreSQL schemas for all business entities
  - Row-Level Security (RLS) policies enforced at DB level
  - Triggers for automatic reference number generation
  - Functions for stock calculation queries
  - Supabase Realtime channels for live subscriptions

LAYER 4: AUTOMATION ENGINE (Background Workers)
  - BullMQ worker processes running on Railway
  - Cron-based rule evaluation (every 15 minutes)
  - Event-driven triggers for stock threshold breaches
  - Email/webhook notification dispatch

LAYER 5: AI SERVICES
  - FAISS vector index service (Python FastAPI on Render)
  - Grok API integration for natural language processing
  - Prophet/scikit-learn prediction service (Python FastAPI on Render)
  - Embedding generation pipeline for new inventory records

6.3 SCREEN: PRODUCT MANAGEMENT
================================================================================

ROUTE: /products

PURPOSE: Define and manage the product catalog. Products are the items tracked
in inventory but do not themselves hold stock — stock is tracked in inventory_stock.

--------------------------------------------------------------------------------
LAYOUT
--------------------------------------------------------------------------------

  [Header: "Products" | [+ Add Product] button | Search bar]
  [Filter bar: Category | Unit of Measure]
  [Products Table]

--------------------------------------------------------------------------------
PRODUCTS TABLE COLUMNS
--------------------------------------------------------------------------------

  Column 1:  Product Name
  Column 2:  SKU Code
  Column 3:  Category
  Column 4:  Unit of Measure (e.g., pcs, kg, liters)
  Column 5:  Unit Price
  Column 6:  On Hand Quantity (read-only, from inventory_stock)
  Column 7:  Available Quantity (read-only, computed)
  Column 8:  Actions (Edit | Archive)

STOCK QUANTITY DEFINITIONS:
  On Hand Quantity:   Total physical quantity currently in warehouse.
                      Formula: SUM(inventory_stock.on_hand_qty) for this product.

  Available Quantity: Stock available for fulfillment.
                      Formula: On Hand - Qty reserved for pending delivery orders.
                      A product can be On Hand but not Available if it is already
                      allocated to a Waiting delivery order.

--------------------------------------------------------------------------------
ADD / EDIT PRODUCT FORM (Modal or Side Panel)
--------------------------------------------------------------------------------

FIELDS:
  Product Name       text        required, 2–200 chars, unique
  SKU Code           text        required, unique, auto-generated or user-entered
                                 Format: can be alphanumeric, e.g. SP-00123
  Category           select      required, from predefined categories table
                                 User can also create new category inline
  Unit of Measure    select      pcs / kg / liters / boxes / meters / other
  Unit Price         decimal     required, >= 0, currency format
  Reorder Level      integer     optional, default 0. Used for low-stock alerts
  Description        textarea    optional, product description
  Is Active          toggle      default ON; archiving sets to false

BUTTONS:
  [Save]     [Cancel]

VALIDATION:
  - SKU must be unique across all products
  - Unit Price must be a valid decimal
  - Product Name cannot be blank

BACKEND LOGIC:
  POST /api/products        → INSERT into products table
  PUT  /api/products/:id    → UPDATE product record
  GET  /api/products        → SELECT with pagination, search, filter
  DELETE /api/products/:id  → Soft delete (set is_active = false)

SKU AUTO-GENERATION:
  If user leaves SKU blank, generate using: SP-{YEAR}{5-digit sequence}
  Example: SP-2024-00001

6.4 SCREEN: INVENTORY STOCK VIEW
================================================================================

ROUTE: /inventory

PURPOSE: Show current stock levels for all products and allow manual stock
adjustments with full audit trail.

--------------------------------------------------------------------------------
LAYOUT
--------------------------------------------------------------------------------

  [Header: "Inventory" | [Adjust Stock] button | Search | Filter by Warehouse]
  [Inventory Table]

--------------------------------------------------------------------------------
INVENTORY TABLE COLUMNS
--------------------------------------------------------------------------------

  Column 1:  Product Name
  Column 2:  SKU
  Column 3:  Warehouse
  Column 4:  Location
  Column 5:  Unit Cost
  Column 6:  On Hand Quantity
  Column 7:  Available Quantity
  Column 8:  Reserved Quantity  (allocated to pending deliveries)
  Column 9:  Status (Normal / Low Stock / Out of Stock)
  Column 10: Actions [Adjust]

STATUS LOGIC:
  Out of Stock:  on_hand_qty = 0
  Low Stock:     on_hand_qty > 0 AND on_hand_qty <= reorder_level
  Normal:        on_hand_qty > reorder_level

--------------------------------------------------------------------------------
STOCK ADJUSTMENT MODAL
--------------------------------------------------------------------------------

Triggered by: [Adjust] button per row

FIELDS:
  Product        display only (pre-filled)
  Current Qty    display only (pre-filled)
  Adjustment Type:
                 select: Add / Remove / Set Absolute Value
  Quantity       integer, required, > 0
  Reason         select:
                   - Physical count correction
                   - Damage / Loss
                   - Initial stock entry
                   - Supplier correction
                   - Other
  Notes          textarea, optional

BUTTONS:
  [Confirm Adjustment]  [Cancel]

BACKEND LOGIC:
  POST /api/inventory/adjust
    Body: { product_id, warehouse_id, location_id, adjustment_type, qty, reason, notes }
    
    Server logic:
      1. Compute new quantity based on adjustment_type
      2. UPDATE inventory_stock.on_hand_qty
      3. INSERT into stock_moves:
           {
             reference: "ADJ/YYYY/NNNN",
             move_type: "adjustment",
             product_id,
             from_location_id: null,
             to_location_id: location_id,
             quantity: adjustment_qty,
             reason,
             notes,
             performed_by: current_user_id
           }
      4. Return updated inventory record

CONSTRAINTS:
  - Remove adjustments cannot reduce quantity below 0
  - All adjustments create a stock_moves record (full audit trail)

6.8 SCREEN: DELIVERY ORDERS (OUTGOING INVENTORY)
================================================================================

ROUTE: /operations/deliveries

PURPOSE: Manage outgoing shipments to customers or internal transfers. Validating
a delivery decreases inventory stock.

--------------------------------------------------------------------------------
LAYOUT
--------------------------------------------------------------------------------

  [Header: "Delivery Orders" | [+ New Delivery] button | Search | Filter by Status]
  [Deliveries Table]

TABLE COLUMNS:
  Reference         (WH/OUT/0001)
  Delivery Address  / Customer
  Responsible User
  Scheduled Date
  Status            (Draft / Waiting / Ready / Done)
  Total Items
  Actions           [View] [Edit] [Validate] [Cancel]

STATUS LIFECYCLE:
  DRAFT  →  WAITING  →  READY  →  DONE

  DRAFT:    Initial state. Can be edited. No stock reservation.
  WAITING:  Stock reserved (quantity subtracted from Available, not On Hand).
            If stock insufficient when moving to Waiting, alert is shown.
  READY:    Goods picked and ready for shipment.
  DONE:     Goods shipped. On Hand quantity decreased.

STOCK RESERVATION MECHANISM:
  When status moves to WAITING:
    For each line: decrease inventory_stock.available_qty by ordered_qty
    (available_qty = on_hand_qty - reserved_qty)
    reserved_qty += ordered_qty in a separate reservations table or computed field

  When status moves to DONE:
    For each line: decrease inventory_stock.on_hand_qty by shipped_qty
    Release reservation

INSUFFICIENT STOCK ALERT:
  When creating or transitioning to WAITING:
    If any product line has ordered_qty > available_qty:
      Show warning: "Insufficient stock for [Product Name].
                     Available: X units. Ordered: Y units.
                     Proceeding will create a negative reservation."
      Options: [Proceed Anyway] [Adjust Qty] [Cancel]

--------------------------------------------------------------------------------
DELIVERY DETAIL / CREATE FORM
--------------------------------------------------------------------------------

  Source Location    select from locations
  Notes              textarea, optional

PRODUCT LINE TABLE:
  Product            select from products (searchable)

  Shipped Qty        integer, editable only at DONE transition
  Available Qty      display only (real-time from inventory)
  Unit Price         decimal, from product defaults

  [+ Add Product Line]  [Remove Line]

BUTTONS:
  [Save Draft]         → status = Draft
  [Confirm Order]      → status = Waiting (reserves stock)
  [Mark Ready]         → status = Ready
  [Validate Delivery]  → status = Done, triggers stock decrease
  [Cancel Delivery]    → Cancelled, releases reservations

VALIDATE DELIVERY LOGIC:
  1. Confirm dialog: "Validating will reduce stock for X products."
  2. For each line:
     a. Check on_hand_qty >= shipped_qty; if not, block or warn
     b. UPDATE inventory_stock: on_hand_qty -= shipped_qty
     c. Release reservation
     d. INSERT stock_moves:
          {
            reference: delivery.reference,
            move_type: "delivery",
            product_id,
            from_location_id: source_location_id,
            to_location_id: null,  (external / customer)
            quantity: shipped_qty * -1,  (negative = outbound)
            performed_by: current_user_id,
            delivery_id: delivery.id
          }
  3. UPDATE deliveries.status = 'done'

RULE EXECUTION ENGINE (BACKGROUND WORKER)

TECHNOLOGY: BullMQ with Redis on Railway

WORKER PROCESS:
  1. BullMQ repeatable job: runs every 15 minutes
  2. Worker fetches all active rules from automation_rules table
  3. For each rule:
     a. Parse condition_config JSON
     b. Execute appropriate SQL query against Supabase to evaluate condition

7. DEMAND PREDICTION MODULE
================================================================================

PURPOSE: Use machine learning to forecast future inventory demand per product,
enabling proactive purchasing and stock planning.

--------------------------------------------------------------------------------
7.1 DATA PIPELINE
--------------------------------------------------------------------------------

INPUT DATA:
  Source 1: deliveries table + delivery_items
    - Historical outgoing quantities per product per date
    - Each validated delivery is a demand data point

  Source 2: stock_moves table
    - Filtered to move_type = 'delivery' or 'adjustment'
    - Provides granular movement history

DATA PREPARATION (Python / pandas):
  1. Query Supabase for all stock_moves of type 'delivery' per product
  2. Group by product_id and date (daily aggregation)
  3. Fill missing dates with 0 demand
  4. Output: time series DataFrame per product:
       date | product_id | demand_quantity

MINIMUM DATA REQUIREMENT:
  Prophet requires at least 2 periods of data to make meaningful forecasts.
  For daily forecasting: minimum 30 days of history recommended.
  If insufficient data: return message "Insufficient history for forecast"

--------------------------------------------------------------------------------
7.2 FORECASTING MODELS
--------------------------------------------------------------------------------

PRIMARY MODEL: Facebook Prophet
  - Handles seasonality (weekly, yearly patterns)
  - Robust to missing data
  - Provides uncertainty intervals (confidence bounds)
  - Trained per product

  Prophet config:
    m = Prophet(
        changepoint_prior_scale=0.05,
        seasonality_mode='multiplicative',
        weekly_seasonality=True,
        yearly_seasonality=True
    )
    m.add_country_holidays(country_name='IN')   # India holidays
    m.fit(df)
    future = m.make_future_dataframe(periods=30)  # 30-day forecast
    forecast = m.predict(future)

FALLBACK MODEL: scikit-learn Linear Regression
  Used when Prophet insufficient (< 30 data points)
    - Features: day_of_week, week_of_year, month, lag_7day_avg
    - Target: demand_quantity
    - Simple trend extrapolation for new products

--------------------------------------------------------------------------------
7.3 PREDICTION SERVICE API
--------------------------------------------------------------------------------

Python FastAPI service on Render.

ENDPOINTS:

  GET /predict/product/{product_id}?days=30
    Returns:
    {
      product_id: "xxx",
      product_name: "Steel Rods",
      forecast: [
        { date: "2024-02-01", predicted_demand: 45.2, lower_bound: 30, upper_bound: 62 },
        { date: "2024-02-02", predicted_demand: 42.1, lower_bound: 28, upper_bound: 59 },
        ...
      ],
      model_used: "prophet",
      confidence: "high",
      training_data_points: 180
    }

  GET /predict/all-products?days=30
    Returns forecast for all products (batch computation)

  POST /predict/retrain
    Admin endpoint: triggers model retraining on latest data
    Retraining scheduled nightly via cron on Render

--------------------------------------------------------------------------------
7.4 FRONTEND INTEGRATION
--------------------------------------------------------------------------------

Location: Dashboard → "Demand Forecast" tab, or Product detail page

Display:
  - Line chart: Actual historical demand (solid line) + Predicted demand (dashed line)
  - Confidence band shaded area (lower_bound to upper_bound)
  - Recommendation text: "Based on forecast, reorder [X units] by [date]"
  - Reorder trigger alert if forecasted demand > current available stock within 14 days

TABLE: inventory_stock
--------------------------------------------------------------------------------
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid()
  product_id      UUID NOT NULL REFERENCES products(id) ON DELETE RESTRICT
  warehouse_id    UUID NOT NULL REFERENCES warehouses(id) ON DELETE RESTRICT
  location_id     UUID REFERENCES locations(id) ON DELETE SET NULL
  on_hand_qty     INTEGER NOT NULL DEFAULT 0 CHECK (on_hand_qty >= 0)
  reserved_qty    INTEGER NOT NULL DEFAULT 0 CHECK (reserved_qty >= 0)
                  -- reserved = allocated to pending delivery orders
  unit_cost       DECIMAL(12,2)                     -- actual cost of current stock
  last_movement_at TIMESTAMPTZ
  created_at      TIMESTAMPTZ DEFAULT NOW()
  updated_at      TIMESTAMPTZ DEFAULT NOW()

  -- Computed: available_qty = on_hand_qty - reserved_qty (in application layer or view)

  CONSTRAINT uq_stock_location UNIQUE (product_id, warehouse_id, location_id)

VIEWS:
  CREATE VIEW inventory_with_availability AS
  SELECT
    s.*,
    p.name AS product_name, p.sku, p.reorder_level,
    (s.on_hand_qty - s.reserved_qty) AS available_qty,
    l.name AS location_name, l.code AS location_code,
    w.name AS warehouse_name
  FROM inventory_stock s
  JOIN products p ON p.id = s.product_id
  LEFT JOIN locations l ON l.id = s.location_id
  JOIN warehouses w ON w.id = s.warehouse_id;

ROW-LEVEL SECURITY POLICIES
--------------------------------------------------------------------------------

All tables have RLS enabled in Supabase.

Example policy for inventory_stock:
  -- Users can only see stock for their organization (future multi-tenant)
  -- Current implementation: all authenticated users can read
  CREATE POLICY "authenticated_read_inventory"
    ON inventory_stock FOR SELECT
    USING (auth.role() = 'authenticated');

  -- Only managers and admins can modify stock
  CREATE POLICY "manager_update_inventory"
    ON inventory_stock FOR UPDATE
    USING (
      EXISTS (
        SELECT 1 FROM users
        WHERE auth_id = auth.uid()
        AND role IN ('admin', 'manager')
      )
    );

10. SECURITY DESIGN
================================================================================

--------------------------------------------------------------------------------
10.1 AUTHENTICATION
--------------------------------------------------------------------------------

MECHANISM: Supabase Auth (JWT-based)
  - User signs in → Supabase returns access_token (JWT) and refresh_token
  - Access token stored in memory (not localStorage for security)
  - Refresh token stored in HTTP-only cookie (inaccessible to JS)
  - JWT expires in 1 hour; refreshed automatically
  - On logout: tokens invalidated server-side via Supabase

SESSION MANAGEMENT:
  - Next.js middleware validates JWT on every protected route
  - Supabase createServerClient used in middleware to verify session
  - Unauthenticated requests to /api/* return 401
  - Unauthenticated page access redirects to /login

--------------------------------------------------------------------------------
10.2 ROLE-BASED ACCESS CONTROL (RBAC)
--------------------------------------------------------------------------------

ROLES:
  admin:    Full access. Can manage users, warehouses, automation, settings.
  manager:  Can create/edit products, receipts, deliveries, inventory adjustments.
            Can configure automation rules. Cannot manage users.
  staff:    Read access to all screens. Can validate receipts and deliveries.
            Cannot create products or configure automation.

ROLE ENFORCEMENT:

  Layer 1 — API Route Middleware:
    Every API route checks user role from JWT claims.
    Helper function:
      function requireRole(allowedRoles: string[]) {
        // Verify JWT, extract role, check against allowedRoles
        // Return 403 if insufficient permission
      }

  Layer 2 — Supabase RLS:
    PostgreSQL Row-Level Security policies enforce data access at DB level.
    Even if API middleware is bypassed, RLS provides defense-in-depth.

  Layer 3 — Frontend UI:
    Buttons and nav items hidden based on user role.
    Role stored in React context from /api/auth/me response.
    Role checks for UI: useRole() hook returns current role.

--------------------------------------------------------------------------------
10.3 AI COMMAND CONFIRMATION (MANDATORY SAFEGUARD)
--------------------------------------------------------------------------------

PRINCIPLE: No AI-initiated action executes without explicit human approval.

IMPLEMENTATION:
  Step 1: AI parses command and returns preview (never executes immediately)
  Step 2: UI renders confirmation dialog with full details of what will happen
  Step 3: User must click [Confirm & Execute] button
  Step 4: Only then does /api/ai/command/execute get called
  Step 5: Both the command_log_id and user session are validated server-side

ADDITIONAL SAFEGUARDS:
  - Command execution endpoint verifies the log entry exists and is in 'pending' state
  - Execution logs are written before and after action
  - Repeated failed execution attempts are rate-limited (5 per minute per user)
  - High-risk operations (delete, large quantity adjustments) require double confirmation

--------------------------------------------------------------------------------
10.4 AUDIT LOGGING
--------------------------------------------------------------------------------

ALL STATE-CHANGING OPERATIONS are logged:

  1. Inventory adjustments → stock_moves table (immutable)
  2. Receipt validations → stock_moves + receipts.done_date
  3. Delivery validations → stock_moves + deliveries.done_date
  4. AI command executions → ai_action_logs
  5. Automation rule triggers → ai_action_logs
  6. User login/logout → Supabase Auth logs
  7. Settings changes → audit_logs table (key-value change log)

AUDIT LOG RETENTION:
  - stock_moves: permanent (append-only, never deleted)
  - ai_action_logs: 1 year retention
  - Supabase Auth logs: per Supabase plan (90 days minimum)

--------------------------------------------------------------------------------
10.5 ADDITIONAL SECURITY MEASURES
--------------------------------------------------------------------------------

INPUT VALIDATION:
  - All API inputs validated with Zod schemas server-side
  - SQL injection prevention via parameterized queries (Supabase client)
  - XSS prevention via React's built-in escaping + Content Security Policy headers

RATE LIMITING:
  - API routes: 100 requests per minute per user
  - AI endpoints: 20 requests per minute per user
  - Auth endpoints: 10 attempts per minute per IP (Supabase built-in)

CORS:
  - next.config.js: allowed origins restricted to vercel.app domain and custom domain
  - API routes: explicit CORS headers

SECRETS MANAGEMENT:
  - All API keys in environment variables (never in code)
  - Vercel environment variables for frontend/backend
  - Render environment variables for Python services
  - GROK_API_KEY, SUPABASE_SERVICE_ROLE_KEY never exposed to client

HTTPS:
  - Enforced by Vercel (frontend + API routes)
  - Enforced by Render (Python services)
  - Supabase: HTTPS only

    CREATE SEQUENCE receipt_seq START 1;
    CREATE OR REPLACE FUNCTION set_receipt_reference()
    RETURNS TRIGGER AS $$
    BEGIN
      NEW.reference := 'WH/IN/' || LPAD(nextval('receipt_seq')::text, 4, '0');
      RETURN NEW;
    END;
    $$ LANGUAGE plpgsql;
    CREATE TRIGGER receipt_reference_trigger
    BEFORE INSERT ON receipts
    FOR EACH ROW WHEN (NEW.reference IS NULL)
    EXECUTE FUNCTION set_receipt_reference();
